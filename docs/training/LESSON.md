# Section 1 — Source-Code Security Scanning: SAST & DAST (12 min)

## Slide 3 — Section divider

## Slide 4 — Why scan source code at all?

- Vulnerabilities are cheapest to fix **before** they ship — shift left.
- Manual review doesn't scale to every commit, repo, and language.
- Scanners give breadth; humans give judgement — we want both.
- Core tension: catch everything ⇒ cry wolf constantly.

*Speaker notes:* A bug caught in review costs minutes; in production after a
breach, orders of magnitude more. But developers ignore tools that bury one real
bug under fifty false alarms. VulnHunterX is a direct response to that
alert-fatigue problem.

## Slide 5 — SAST vs DAST in one picture *(table)*

| | **SAST** (static) | **DAST** (dynamic) |
|---|---|---|
| Sees | Code/bytecode at rest | A running application |
| Runs the app? | No | Yes |
| Coverage | Every path, even unreached | Only paths exercised at runtime |
| Finds | Injection, memory bugs, config, secrets | Auth flaws, real exploitability |
| Weakness | False positives | False negatives (misses unreached code) |
| In the SDLC | Commit / PR / IDE | Staging / QA / pre-prod |

*Speaker notes:* SAST reads code like a pedantic reviewer — sees everything, can't
prove reachability, over-reports. DAST throws real traffic at a live app —
exploitable but only covers what it hit. Complementary. VulnHunterX is a SAST tool
(its optional fuzzing stage adds DAST-like runtime confirmation for C/C++ — out of
scope today).

## Slide 6 — The SAST false-positive problem

- A raw SAST run can emit hundreds to thousands of findings.
- Many are unreachable, already mitigated, or the wrong CWE class.
- Hand triage is slow and demoralising → tools get abandoned.
- The opportunity: automate the triage **reasoning**, not just detection.

*Speaker notes:* Detection is mostly solved — CodeQL and Semgrep find candidates
well. The expensive, unsolved part is deciding which are real. VulnHunterX inserts
an LLM as a first-pass triager. Hold the idea: *detection finds candidates,
verification decides.*

# Section 2 — How CodeQL & Semgrep Scan (24 min)

## Slide 7 — Section divider

## Slide 8 — Two philosophies of static analysis *(two-column diagram)*

- **Pattern matching · Semgrep** — "does this code *look like* a known
  bad shape?" Parses each file to an AST, matches a written pattern. File-local,
  fast, no build. Flow: `source file → AST → pattern match → finding`.
- **Semantic / data-flow · CodeQL** — "can tainted data actually *reach* a sink?"
  Builds a queryable model of the whole program. Cross-function, type-aware, needs
  a build. Flow: `build → relational DB → taint query → finding`.
- *Footer:* different questions, not rivals — both emit SARIF.

*Speaker notes:* The reframing: not two competing tools, two different
*questions*. Pattern matching asks "does this look wrong?"; semantic analysis asks
"can bad data get here?" The shared SARIF output is just the bridge. VulnHunterX
runs both and reconciles.

## Slide 9 — CodeQL: code becomes a database

- Build the code under a tracer → a relational DB of the program (functions,
  expressions, types, control & data flow).
- Query it with `.ql` queries — an SQL-like, OO language.
- Built-in suites: `security-extended`, `security-and-quality`.
- Output: SARIF with ruleId, location, severity, data-flow path.
- **Needs to observe the build** for compiled langs (C/C++/Java/Go/C#).

*Speaker notes:* Key fact: CodeQL must observe the build for compiled languages —
a broken build collapses coverage (remember for limitations). Once the DB exists,
"flow from argv to system()" runs across function boundaries — what pattern tools
can't do.

## Slide 10 — CodeQL: the extraction → query pipeline *(flowchart)*

`Source code → Build tracer → Relational DB (AST+CFG+DFG+types) → .ql query
(suite + custom) → SARIF (ruleId, location, data-flow path)`

Two phases: **extraction** builds the DB once (slow, needs the build); then **any
number of queries** run over it cheaply. Break the build → incomplete DB →
memory-safety recall collapses.

*Speaker notes:* The expensive, fragile step is the build tracer + extraction. Once
the DB exists, queries are cheap and many. A broken build is catastrophic for
CodeQL: no DB, no semantic findings. The workshop repos were chosen because they
build reliably.

## Slide 11 — Anatomy of a `.ql` query (taint tracking) *(code)*

```ql
import cpp
import semmle.code.cpp.dataflow.TaintTracking

from DataFlow::Node source, DataFlow::Node sink
where
    source.asExpr() instanceof ArgvAccess and          // tainted input
    sink.asExpr() = aSystemCall().getArgument(0) and    // dangerous op
    TaintTracking::flowPath(source, sink)               // a real path?
select sink, "argv reaches system() with no sanitizer"  // CWE-78
```

*Speaker notes:* You don't write CodeQL to use VulnHunterX, but one query
demystifies it. Note the three ingredients from the AST section: a SOURCE, a SINK,
and `flowPath()` asking whether tainted data actually connects them. Custom rules
under `--profile full` are files like this.

## Slide 12 — Semgrep: patterns over syntax *(code)*

```yaml
rules:
  - id: dangerous-system-call
    patterns:
      - pattern: system($CMD)
    message: "Command injection risk if $CMD is tainted"
    metadata:
      cwe: ["CWE-78"]        # used later to route guided questions
```

No build required — reads source directly. Registry packs: `p/security-audit`,
`p/gosec`, `p/owasp-top-ten`.

*Speaker notes:* `metadata.cwe` matters later for routing. Offline caveat: `p/...`
registry packs need semgrep.dev. For reliable offline coverage, the in-repo
`config/semgrep-custom/<lang>.yaml` rules load under `--profile full`.

## Slide 13 — How pattern matching works *(table)*

| Construct | Means | Example |
|---|---|---|
| `$X`, `$CMD` | Metavariable — matches any expression | `system($CMD)` |
| `...` | Ellipsis — matches any sequence | `foo(..., $X)` |
| `pattern` | Must match | `exec($C)` |
| `pattern-not` | Excludes a safe form (cuts FPs) | `exec("static")` |
| `pattern-either` | Any of several (OR) | `MD5(...) | SHA1(...)` |
| `metavariable-pattern` | Constrain what a `$VAR` may be | `$URL ~ user input` |

Matching is on the **AST, not raw text** — whitespace/comments/formatting don't
matter.

*Speaker notes:* The power and the limit in one slide. Metavariables + ellipsis
generalise across formatting; `pattern-not` suppresses the safe form to cut FPs.
But it's still file-local and syntactic — can't follow tainted data across
functions like CodeQL's `flowPath()`. That gap is why LLM verification earns its
keep.

## Slide 14 — SARIF: the common currency

- **S**tatic **A**nalysis **R**esults **I**nterchange **F**ormat (JSON).
- Each finding: `ruleId`, file+line, severity, message, dataflow, CWE tags.
- VulnHunterX discovers **all** `*.sarif` files → one `[Finding]` list.
- Tool-agnostic downstream → enables cross-tool reconciliation.

*Speaker notes:* Because everything funnels through SARIF, verification doesn't
care which tool produced a finding. It also lets VulnHunterX reconcile when two
tools flag the same line — majority vote for sibling rules, safe FP→NMD downgrade
across CWE classes.

## Slide 15 — SARIF anatomy — the fields the parser reads *(code)*

```json
{ "runs": [ {
  "tool": { "driver": { "name": "CodeQL" } },
  "results": [ {
      "ruleId": "cpp/use-after-free",          // → routes guided questions
      "level": "error",                         // severity
      "message": { "text": "freed pointer reused" },
      "locations": [ { "...": "imgRead.c", "startLine": 67 } ],
      "codeFlows": [ "..." ],                    // the data-flow path
      "properties": { "cwe": ["CWE-416"] }       // → CWE fallback routing
  } ] } ] }
```

*Speaker notes:* Point out the five fields that matter downstream: `ruleId` (exact
routing), `level`, `location` (file+line the LLM anchors on), `codeFlows` (the
data-flow path), and `properties.cwe` (fallback routing for Semgrep). This is the
literal hand-off from detection to verification.

## Slide 16 — Strengths & weaknesses, side by side *(table)*

| | CodeQL | Semgrep |
|---|---|---|
| Setup | Needs build (compiled langs) | None — reads source |
| Depth | Cross-function taint, CFG/DFG | File-local pattern / AST |
| Speed | Slower (build + DB) | Fast |
| Best at | Injection, memory-safety flow | Config, secrets, syntactic bugs |
| Offline | Yes (queries are local) | Custom yes; `p/` packs need network |

*Speaker notes:* Use both. `--tool both` runs CodeQL + Semgrep; `--profile full` adds
in-repo custom rules. The redundancy is deliberate — different engines miss
different bugs, and agreement is a useful triage signal.

# Section 3 — AST, Control Flow & Data Flow (22 min)

## Slide 17 — Section divider

## Slide 18 — What is an AST?

- Abstract Syntax Tree — the parse tree of code as structured nodes.
- `x = a + b * 2` → assignment ▸ `+` ▸ `*` … structure, not text.
- Scanners match on the tree, not characters → robust to whitespace/names.
- `system(cmd)` ≡ `system( cmd )` ≡ `system(/*x*/cmd)` — same tree.

*Speaker notes:* Why not regex? Those three forms are the same program but
different text. An AST normalises that away. Semgrep and tree-sitter work on ASTs;
CodeQL builds a richer model on top.

## Slide 19 — An AST, drawn *(tree diagram)*

A node-and-edge tree for `x = a + b * 2`: root `=` with children `x` and `+`; `+`
has children `a` and `*`; `*` has children `b` and `2`.

*Speaker notes:* Operator precedence isn't re-derived each time — it's baked into
the shape: `*` sits below `+`. A pattern like `$X = $A + $B` matches the top two
levels regardless of spacing or names. This is the structure everything builds on.

## Slide 20 — From AST to control flow & data flow

- **Control-Flow Graph (CFG):** the order statements *can* execute.
- **Data-Flow Graph / taint:** how values move between variables.
- Security questions are flow questions: **SOURCE → SANITIZER → SINK**.
- A vuln = source → sink flow with **no adequate sanitizer**.

*Speaker notes:* This source → sanitizer → sink model is the single most useful
idea in the talk. Taint tracking = "is there a path on the data-flow graph from
source to sink?" The guided questions push the LLM to anchor on exactly this.

## Slide 21 — Control flow: which paths can execute *(CFG diagram)*

`read len → if (len > MAX)?` branches to **return error** (safe, `len ≤ MAX`) or
**`memcpy(dst, src, len)`** (sink — overflow if `len > MAX`).

The CFG is the set of paths control *can* take. The bug exists on **one branch**,
only if reachable with a bad `len`. Proving reachability is what SAST struggles
with (→ FPs).

*Speaker notes:* The vulnerability lives on exactly one edge. A pattern tool sees
the `memcpy` and flags it; it can't easily tell whether the guard makes the bad
branch unreachable. That reachability question is the heart of false positives —
and of what verification reasons about.

## Slide 22 — Data flow: does tainted input reach a sink? *(taint diagram)*

**SOURCE** (argv · request · file) → **SANITIZER** (validate/escape/bounds-check)
→ **SINK** (`system()` · `memcpy()`). Missing/inadequate sanitizer ⇒ source reaches
sink = **vulnerability**.

*Speaker notes:* The one diagram to remember. Every guided-question set is really
pushing the LLM to fill in these three boxes for the specific finding: find the
source, find the sink, check what's in between.

## Slide 23 — CodeQL vs tree-sitter for building the AST *(table)*

| | CodeQL | tree-sitter |
|---|---|---|
| Model | Semantic: AST+CFG+DFG+types | Syntactic AST only |
| Needs build? | Yes (compiled langs) | No — parses source |
| Speed | Slower | Very fast |
| Depth | Cross-function, type-aware | File-local, structural |
| In VulnHunterX | Default context backend | Fast fallback; all 8 languages |

*Speaker notes:* Both are context-extraction backends
(`--backend {auto,codeql,treesitter}`). `auto` picks CodeQL when a DB exists, else
tree-sitter.

## Slide 24 — tree-sitter — the fast, build-free parser *(cards)*

- **Incremental parser** — re-parses only edited regions.
- **Per-language grammars** — all 8 languages covered.
- **No build needed** — works even when the build is broken.
- **Error-tolerant** — parses partial / WIP code.
- **Syntactic only** — no types, no cross-function data-flow.
- **Role:** the `treesitter` context backend, fallback when no CodeQL DB.

*Speaker notes:* tree-sitter powers editor syntax highlighting. Superpowers for us:
no build, all languages, error-tolerant. Limit: purely syntactic — gives the
enclosing function and structure, not types or taint. Hence the fallback role.

## Slide 25 — Doesn't Semgrep already build an AST? *(table)*

| | Semgrep's AST | tree-sitter (in VulnHunterX) |
|---|---|---|
| Whose engine | Internal to Semgrep | VulnHunterX's ContextExtractor |
| Purpose | **Detect** findings (pattern match) | **Extract context** for the LLM |
| Reusable? | No — private to the matcher | Yes — query for functions, callers, structs |
| Scope | The rule's match site | Whole codebase → context CSVs, all 8 langs |
| When used | Only if Semgrep runs & fires | Any finding (incl. CodeQL-only), build-free |

*Different ASTs, different jobs — detection vs context.*

*Speaker notes:* Yes, Semgrep parses to an AST — but that tree is private to its
pattern matcher and exists only to **detect**. VulnHunterX needs its **own**
queryable AST to **extract context** (enclosing function, callers, struct fields)
that grounds the LLM. Findings often come from **CodeQL alone** (Semgrep may not
run or fire), so the tool can't depend on Semgrep's internal tree. CodeQL is the
richer semantic backend; tree-sitter is the fast, build-free fallback covering
all 8 languages (`src/vuln_hunter_x/context/treesitter_extractor.py`).

## Slide 26 — AST context: from repo to LLM prompt *(flowchart)*

`Repo → Extraction (CodeQL/tree-sitter) → Context CSVs (functions, callers,
structs, free_sites…) → ContextProvider (look-up + cache) → Code slice (enclosing
fn + requested context) → LLM prompt (grounded)`

The model never sees the 100k-line repo — it sees a tight, AST-derived slice
relevant to the finding. The single most important defence against hallucination.

*Speaker notes:* The bridge from Part 3 into the VulnHunterX sections. Stage 1
turns the repo into CSVs; the ContextProvider assembles a small relevant slice and
only that goes into the prompt. Grounding in real extracted code keeps verdicts
honest. Full diagram in `docs/context-extraction-flow.md`.

## Slide 27 — The context types the LLM can pull *(table)*

| Context type | Answers | When |
|---|---|---|
| function / method | What does this code do? | pre-fetched |
| caller / all_callers | Who calls it? Where does input come from? | pre-fetched |
| callees | What sinks does it call? | pre-fetched |
| struct / class | Field types & layout? | reactive |
| global / macro | This constant / expansion? | reactive |
| typedef / enum | Underlying type / values? | reactive |
| free_sites *(C/C++)* | Where is this pointer freed? | reactive |
| destructor *(C/C++)* | What cleanup runs on this type? | reactive |
| field_writes *(C/C++)* | Where is this field written? | reactive |

*Speaker notes:* 11 context types by phase. Pre-fetched = cheap, function-keyed,
sent up front. Reactive = only when the model asks in a Needs-More-Data turn. The
C/C++ memory-safety trio powers use-after-free / double-free verification.

# Section 4 — LLM Vulnerability Verification: Pros & Cons (14 min)

## Slide 28 — Section divider

## Slide 29 — Why add an LLM after the scanner?

- Scanner: "line 58 might be a heap overflow." It can't add "…and it's reachable
  with attacker-controlled length → real."
- An LLM, given code + context, can reason about reachability & sanitizers.
- Goal: noisy candidate list → ranked, explained, triaged report.
- It does **not** detect new bugs — it judges the ones SAST produced.

*Speaker notes:* The LLM does first-pass human-style triage at machine speed and
scale. Explicitly not the detector — it never invents findings.

## Slide 30 — What we actually ask the LLM *(cards)*

- **Verdict:** TRUE_POSITIVE · FALSE_POSITIVE · NEEDS_MORE_DATA
- **Confidence:** high / medium / low → 0.85 / 0.6 / 0.3
- **Reasoning:** anchored to specific lines
- **Answers:** to the rule's guided questions
- **`context_needed`:** extra code it wants next → drives the multi-turn loop

*Speaker notes:* NEEDS_MORE_DATA is the interesting one — instead of guessing, the
model asks for the caller or struct; the engine fetches it and re-asks.

## Slide 31 — Pros of LLM verification

- Cuts false-positive noise — the headline benefit.
- Explains each verdict → faster triage & learning.
- Fills context gaps SAST leaves (intent, framework conventions).
- Language-flexible — one reasoning engine across C, Python, Java, JS, Go…

*Speaker notes:* The explanation is underrated. "the pointer freed at 59 is
dereferenced at 67 when size1 % 3 == 0" teaches a junior dev to spot
use-after-free. The tool teaches while it triages.

## Slide 32 — Cons & risks of LLM verification

- Hallucination — can assert a sanitizer or flow that isn't there.
- Non-determinism — same finding, different runs.
- Cost & latency — tokens and time per finding, especially multi-turn.
- Prompt injection — the code under review is attacker-influenced text.
- Overconfidence — fluent prose ≠ correct reasoning.

*Speaker notes:* Be honest — engineers respect it. None are solved perfectly;
they're mitigated (next slide). Assistant, not oracle.

## Slide 33 — How VulnHunterX mitigates the cons *(table)*

| Risk | Mitigation |
|---|---|
| Hallucination | Grounding — CSV/AST context; questions demand line citations |
| Weak/uncited claims | Confidence downgrades — vague reasoning auto-demoted to low |
| Premature verdicts | `min_iterations` on high-stakes CWEs (CWE-416, CWE-200) |
| Non-determinism | Temperature 0.2, structured JSON, tool reconciliation |
| Single-tool error | Cross-tool reconciliation (CodeQL vs Semgrep) |

*Speaker notes:* Grounding and structure beat free-form prompting. That's the
engineering behind a reliable pipeline.

# Section 5 — VulnHunterX Without Fuzzing: Architecture & Stages (28 min)

## Slide 34 — Section divider

## Slide 35 — The big picture *(diagram)*

```
SARIF files
  └─► SarifParser → [Finding]
        └─► VerificationEngine
              ├── QuestionsLoader   (rule-specific guided questions)
              ├── ContextProvider   (CSV / AST look-ups)
              └── LLMClient (LiteLLM → OpenAI · Anthropic · Ollama)
                    └─► Verdict (TP | FP | NEEDS_MORE_DATA) + confidence
```

*Speaker notes:* Everything assembles. SAST → SARIF → parser → engine (questions +
context + LLM) → verdict per finding. LiteLLM is the provider abstraction — the
workshop uses Ollama Cloud.

## Slide 36 — The pipeline: Stages 1–4 *(table)*

| Stage | Command | In → Out |
|---|---|---|
| 1 | `prepare` | repo → source + CodeQL DB + context CSVs |
| 2 | `analyze` | DB/source → SARIF |
| 3 | `verify` | SARIF + context → verdicts + confidence |
| 4 | `report` | verdicts → Markdown report (EN/VI) |

*(Stages 5–8 add ASan build + fuzzing for C/C++ — out of scope today.)*

*Speaker notes:* Four stages to a report. `scan` runs all four; individual
commands let you re-run one stage. Stage 1 creates the AST context CSVs from
Part 3.

## Slide 37 — The pipeline, as a flow *(flowchart)*

`1·prepare → 2·analyze → 3·verify → 4·report`, with artifacts under
`output/<lang>/<repo>/`: `database/ + context/` → `*.sarif` →
`verification_results/` → `report.md`.

*Speaker notes:* Each arrow is a persisted artifact on disk — stop and resume, or
re-run just verify against the same SARIF with a different model. `scan` is the
whole chain in one command.

## Slide 38 — Rule profiles: dialing coverage up *(table)*

| Profile | CodeQL suite | Semgrep packs | Custom |
|---|---|---|---|
| `standard`* | security-extended | auto + per-lang | — |
| `extended` | security-extended | + security-audit, secrets | — |
| `maximum` | security-and-quality | + owasp-top-ten | — |
| `extended-registry` | security-and-quality | 8 universal + per-lang | — |
| `full` | security-and-quality | extended-registry packs | + in-repo custom |

*\* default.*

*Speaker notes:* `--profile full` is the one for offline/self-hosted use — loads
in-repo custom CodeQL + Semgrep, no semgrep.dev. The workshop uses `full`.

## Slide 39 — What the suites & packs actually contain *(table)*

| Suite / pack | What it is |
|---|---|
| CodeQL · security-extended | High-confidence security queries — fewer FPs, the safe default |
| CodeQL · security-and-quality | + maintainability/quality queries — broader, noisier |
| Semgrep · auto | Registry auto-detects languages and applies matching rulesets |
| Semgrep · p/security-audit | Broad cross-language security audit ruleset |
| Semgrep · p/secrets | Hardcoded credentials, API keys, tokens |
| Semgrep · p/owasp-top-ten | Rules grouped by the OWASP Top-10 categories |
| Semgrep · per-language | p/gosec (Go), p/django, p/flask (Py), p/nodejs (JS)… |

*A profile is just a preset bundle of one CodeQL suite + a set of Semgrep packs.*

*Speaker notes:* Two CodeQL suites: security-extended (curated, high-precision)
and security-and-quality (adds maintainability/quality lint — more coverage, more
noise, which is why higher profiles pair it with LLM triage). Semgrep packs: auto
(registry picks per language), p/security-audit (broad sweep), p/secrets
(hardcoded keys), p/owasp-top-ten (OWASP categories), and per-language packs.
`full` layers our in-repo custom rules on top.

## Slide 40 — Custom rules (the `full` profile)

- **Custom CodeQL** — `config/codeql-custom/<lang>/src/*.ql`; `@id` matches a
  question key exactly (e.g. `cpp/use-after-free`).
- **Custom Semgrep** — `config/semgrep-custom/<lang>.yaml`; each sets
  `metadata.cwe` for routing (e.g. `vulnhunterx.cpp.weak-hash`).
- Counts computed from `config/` at build time — never drift.

*Speaker notes:* Custom rules close gaps in framework-aware and C/C++
memory-safety classes, wired to guided questions. `scripts/audit_rule_coverage.py`
shows coverage by rule × CWE × tool.

## Slide 41 — Guided questions

- Per-language YAML banks: `config/prompts/<lang>_questions.yaml`.
- Each rule → evidence-anchored questions the LLM must answer first.
- UAF e.g.: "Where is the pointer allocated? Where freed? Quote the flagged line."
- Also sets `additional_context`, `min_iterations`, snippet window.

*Speaker notes:* The secret sauce against hallucination. The model walks the
evidence: anchor on the flagged line, trace allocation → free → use. A model that
must cite lines can't hand-wave.

## Slide 42 — Anatomy of a guided-question entry *(table)*

| Key | Purpose |
|---|---|
| `short_description` | One-line summary of the class |
| `questions` | Ordered, evidence-anchored questions answered first |
| `context_hint` | Human-readable trace plan (allocation → free → use) |
| `additional_context` | Context types to prefetch / allow requesting |
| `min_iterations` | Floor on rounds before TP/FP (high-stakes CWEs) |
| `snippet_window_lines` | How much code around the flagged line to show |

*Speaker notes:* Each rule maps to a YAML entry shaped like this. Note
`min_iterations` (can't snap-judge a UAF) and `additional_context` (which of the
11 types this class needs). Read by `questions/loader.py`.

## Slide 43 — Real example: py/sql-injection (verbatim)

- Quote the EXACT sink (`cursor.execute` / `session.execute`, line N) and name the
  variable passed to it. Do not paraphrase.
- List EVERY assignment to that variable on each path reaching the sink, with line
  numbers; identify the LAST one that executes.
- Does the value derive from user input (`request.args/form/json/cookies`) or a
  constant? Cite the data-flow chain step by step.
- What specific defence sanitises it? Name it — parameterised
  `execute(sql, params)`, Django ORM `.filter(**kwargs)`… vague "sanitisation" is
  not acceptable.
- If you can't show the tainted value reaching the sink AND the absence of every
  defence, verdict **False Positive**.

*Speaker notes:* The actual questions from `python_questions.yaml`
(`py/sql-injection`), lightly trimmed — chosen to show the same evidence-first
philosophy on a different language and class (taint, not memory safety). It forces
a concrete sink quote, a per-path data-flow trace, a NAMED sanitiser, and a
default-to-FP rule when evidence is missing. `min_iterations=2`. This is the
CWE-89 class that scored F1=100% in the OWASP-Python benchmark.


## Slide 44 — Real example: `cpp/overflow-buffer` (verbatim)

- Where is the DESTINATION buffer declared and what is its exact size?
- Where is the SOURCE buffer/data declared and what is its exact size?
- Do these sizes ever CHANGE via assignment, reallocation, or parameters?
- Any CHECKS on source size, destination size, or copy length BEFORE the copy?
- → The pattern: pin down sizes, track changes, find the missing bound.

*Speaker notes:* A second class to show the pattern generalises: pin the dest size,
the source size, whether either changes, and whether a bound exists before the
copy. Same philosophy — evidence, not vibes.

## Slide 45 — How a finding finds its questions (3-tier routing)

1. **Exact** match on SARIF `ruleId` → `<lang>/<name>` (custom CodeQL hits here).
2. **Normalized / prefix / language-prefix** fallbacks.
3. **CWE-based** fallback via `cwe_question_map` (custom Semgrep routes here).
4. else → generic question template.

*Speaker notes:* CodeQL IDs look like `cpp/use-after-free` (slash) → match a key
directly. Semgrep IDs look like `vulnhunterx.cpp.weak-hash` (dots) → never match
by name, so route by CWE tag. Every finding gets some interrogation.

## Slide 46 — Routing, traced: two findings, two paths *(flowchart)*

- **CodeQL (Tier 1):** `ruleId: cpp/use-after-free` → exact key match →
  `cpp/use-after-free` questions.
- **Semgrep (Tier 3):** `id: vulnhunterx.cpp.weak-hash` → no key (dotted id) →
  `metadata.cwe: CWE-327` → `cwe_question_map` → weak-hash questions.

*Speaker notes:* The CodeQL rule's `@id` *is* the question key, so Tier 1 hits.
The Semgrep dotted id can't match a slash key, so it falls through to Tier 3 and
routes by `metadata.cwe`. Same destination, different door.

## Slide 47 — Context extraction & the ContextProvider

- **Pre-fetched** (keyed on function name): enclosing function, `caller`,
  `callees`, `all_callers`.
- **Reactive** — only when `context_needed` asks: `struct`, `global`, `macro`,
  `typedef`, `enum` + (C/C++) `free_sites`, `destructor`, `field_writes`.
- All reads cached and path-traversal-safe; CSVs come from Stage 1.

*Speaker notes:* Two-phase model: cheap context up front; expensive context lazily
when the model asks. The 11 types map to the evidence guided questions demand.

## Slide 48 — Multi-turn verification loop *(flowchart)*

`LLM analyzes (finding + context)` → decision **verdict?**
- **TP / FP** (once `min_iterations` met) → **Return Verdict** (+ confidence).
- **NEEDS_MORE_DATA** → **Fetch context** (`free_sites`, `caller`, `struct`…) →
  loop back and **re-ask**.

Guards: dedup against already-fetched context · capped at `max_iterations` (10) ·
`min_iterations` floor on high-stakes CWEs · force-decision if it stalls.

*Speaker notes:* The heart of the engine, as a loop. The model emits a verdict +
a `context_needed` list. TP/FP (once min_iterations is met) returns.
NEEDS_MORE_DATA routes to the provider, which fetches the requested context,
appends it, and the LLM re-asks. Requests are deduped so it can't spin forever;
max_iterations caps it; a force-decision turn makes it commit if it stalls.


## Slide 49 — Output: the report

- `verification_results/` — one `verdict_*.json` per finding + `summary.json`.
- `report.md` (EN) and `report_vi.md` (VI) — summary + per-finding detail.
- A Coverage-Limitations caveat documents what SAST can't reach.

*Speaker notes:* The report is what a developer reads. Leads with confident True
Positives, explains each, and is honest about classes the pipeline can't cover.
We'll read one in the workshop.

## Slide 56 — What it does well — and what it misses *(cards)*

**Does well:** syntactic classes (unsafe functions, format string, double-free,
UAF); cross-function data-flow injection; less review load via LLM triage +
reconciliation.

**Misses:** detection misses (integer-overflow chains, divide-by-zero, leaks);
hard tier (C++ lifetime/UB); out of SAST reach (missing authz, CSRF-disabled);
a failed C/C++ build collapses CodeQL recall.

*Speaker notes:* Numbers from `docs/benchmarks/ground-truth-baselines.md`. Two
failure modes: detection misses (no rule fired) vs triage misses (fired but
verified FP/NMD).

## Slide 57 — How we benchmark

- A standalone harness scores VulnHunterX vs raw SAST on ground-truth datasets.
- Datasets: OWASP Benchmark (Java/Python), Juliet C/C++ (64K cases), SecLLMHolmes, RealVuln, OpenVuln/ZeroFalse.
- Approaches: `raw-sast` (no LLM) vs `vulnhunterx` (guided questions + multi-turn) + ablations.
- Metrics: precision · recall · F1 · FP-reduction · NMD rate · tokens/finding · p95 latency · confidence calibration.
- Reproducible: `benchmarks/scripts/run_benchmark.py` & `run_model_matrix.py`.

*Speaker notes:* The harness runs the same datasets through raw-sast (every
finding treated as TP — the baseline) and through the full VulnHunterX pipeline,
scoring both against ground truth, plus a model matrix to compare LLMs. Numbers on
the next two slides are from committed runs under `benchmarks/results/`.

## Slide 58 — Benchmark: false-positive reduction (OWASP-Python) *(table)*

| Approach | Precision | Recall | F1 |
|---|---|---|---|
| raw-sast (baseline, no LLM) | 38.6% | 100.0% | 55.7% |
| vulnhunterx (guided + multi-turn) | 65.9% | 98.0% | 78.8% |

*Model: qwen3-coder:480b (Ollama Cloud) · 500 Python findings.*

*Speaker notes:* The core result. Raw SAST flags everything → recall 100% but
precision only 39% (3 in 5 alerts are false alarms). VulnHunterX lifts precision
to 66% (F1 56%→79%) while keeping recall at 98% — removes most false positives,
keeps almost all real bugs. Per-CWE, SQL-injection (CWE-89) and command-injection
(CWE-78) both hit F1=100%. Confidence is well-calibrated: High verdicts ~81%
accurate vs ~68% for Low.

## Slide 60 — Two kinds of miss — know which you're fighting *(table)*

| | Detection miss | Triage miss |
|---|---|---|
| What | No rule fired at all | Rule fired, but verified FP / NMD |
| Cause | Coverage gap (no query/pattern) | Weak context or LLM error |
| Example | dvcp int-overflow chain, div-by-zero | Real bug downgraded to NMD |
| Remedy | Add custom rule (`--profile full`) | More context, iterations, better model |
| Measure | Recall vs ground truth | Precision of verified verdicts |

*Speaker notes:* Critical distinction for evaluating the tool. Detection miss = a
SAST coverage gap (the LLM never saw it). Triage miss = a verification error.
Completely different fixes; score them separately.

## Slide 61 — Observed coverage on the benchmark repos *(table)*

| Repo (lang) | Detected well | Missed (expected) |
|---|---|---|
| dvcp (C) | double-free, UAF, OOB read/write | int-overflow chain, div-by-zero, leak, exhaustion |
| insecure-coding-examples (C++) | unsafe fns, format string, UAF, double-free, type-confusion | C++ lifetime/UB: dangling string_view, removed memset, RAII |
| insecure-cpp-dojo (C++) | ≈1/14 (Semgrep only) | most — CodeQL build failed → recall collapsed |

*Speaker notes:* Honest numbers from the baselines doc. The dojo row is the

repos.

## Slide 62 — Build health drives memory-safety recall

- CodeQL needs to **observe the build** for C/C++/Java/Go/C#.
- Build fails → DB incomplete → semantic queries find little.
- Semgrep still run (no build) — but only syntactic coverage.
- Practical: get a clean build first; use `--tool both` so something always runs.
- Verify the DB step in `prepare` logs before trusting low finding counts.

*Speaker notes:* The failure is silent — a report with few findings can look like
clean code when CodeQL never built a database. Always check `prepare` succeeded,
and use `--tool both` for a floor of coverage.

## Slide 63 — Cost, latency & best practice *(table)*

| Lever | Why it matters | Recommendation |
|---|---|---|
| `--limit N` | Tokens + time scale with findings | Start with 5–10 while learning |
| model choice | Big models cost more per finding | Cheap/local first, escalate on hard cases |
| `--max-iterations` | Multi-turn = more calls | Default 10; lower for quick passes |
| `-j / --jobs` | Parallelism vs rate limits | Raise to speed up; back off on 429s |
| re-run verify only | Don't rebuild the DB | Iterate on triage cheaply |

*Speaker notes:* LLM triage isn't free — tokens and wall-clock per finding, more
with multi-turn. Use `--limit` and a cheap model, escalate only on hard findings,
re-run just verify. Closes the loop back to "assistant, not oracle."
